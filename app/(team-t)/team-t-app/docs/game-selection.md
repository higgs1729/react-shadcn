# 決定: 9本目のゲームは既製ゲームを流用せず、新規に1本作る

- 日付: 2026-07-21
- 格付け: 不可逆(git履歴へのバイナリ混入) × 影響度 中
- 確信度: 高 — 下げている要因は「見た目の完成度」の合格ラインが未定量であること

## 背景と問い

当初の問いは「無料で入手できる Unity ゲームのうち、できるだけクオリティが高いものを選び、
WebGL ビルドして `public/team-t-app/games/` に追加する」だった。

この問いは以下の理由で成立しないと判明し、立て直した。

1. Unity WebGL の出力は単一 HTML ではなくディレクトリ(`Build/*.wasm`, `*.data` 等)で、
   20〜60MB。既存8本は 8〜72KB の自己完結 HTML 1枚。
2. `lib/team-t-app/games.ts` の `getTeamTGameUrl` は `encodeURIComponent(fileName)` を使うため、
   `dir/index.html` 形式のパスを表現できない。
3. 無料で入手できる「高品質な Unity ゲーム」の大半はビルド済みバイナリ配布で
   プロジェクト(ソース)が付属せず、そもそも再ビルドできない。再配布可否も別問題。
4. 決定的な理由は下記「アプリ契約」。既製ゲームはジャンルを問わず全て契約違反で、
   必ずソース改造が必要になる。

## アプリ契約(この決定を規定した制約)

`components/team-t-app/team-t-game-dialog.tsx` が実装している、ゲーム側が満たすべき契約。

| 契約 | 実装箇所 |
| --- | --- |
| `postMessage({ type: "game:ended", coin })` を親へ送出する | :109 |
| `event.origin !== window.location.origin` は拒否。**同一オリジン必須** | :104 |
| `coin` は `0`〜`maxReward` の整数。`0` は失敗扱い | :110-117 |
| iframe の `load` が10秒以内に発火しないと `load-error` + コイン返却 | :92-95 |
| プレイボタンは親が持ち、コインは**起動前**に減算される | :130-136 |
| `sandbox="allow-scripts allow-same-origin"`。フルスクリーン/ポインタロック不可 | :526 |

この契約から、ゲーム側の必須要件が決まる。

- 単一 HTML ファイル、同一オリジン同梱
- **ゲーム内タイトル画面を持たず、読み込み直後に開始する**(課金済みのため)
- 制限時間内クリア判定をゲーム内で完結させ、結果を `coin` に変換して送出する

## 決定

既製ゲームの流用をやめ、上記契約に最初から適合する単一 HTML ゲームを新規に1本作る。
既製ゲームはどれを選んでも「メニュー除去・尺の短縮・タイマー・クリア判定・postMessage」の
改造が必要で、その総量は新規作成と大差ない一方、リポジトリ肥大とライセンス確認の
コストだけが上乗せされるため。

品質基準は**見た目の完成度**(利用者が明示的に選択)。

この決定は、着手前に「既に良い既製ゲームがないか」を全数探索したうえで再確認している。
探索の記録と、そこで判明したライセンスの一般則は下記「既製ゲーム探索の結果」に残す。

### 実装方針(推奨)

疑似3Dのネオン系ランナー/レース。理由は3点。

- 既存8本のジャンル(パズル3・アクション2・シューティング/RPG2・反射1)に穴がある
- 「x秒生き延びる/走破する」が時間制限クリア判定と自然に噛み合う
- Team T のブラック&ゴールド + ミッドナイト(`#4c2378` / `#b99a5c`)の配色で映える

確定コンセプトは**ネオントンネルランナー**。光るチューブ状コースを高速で疾走し、
障害物を避けて制限時間内に走破する。

### 描画方式と依存の扱い

three.js を使い、ジオメトリはすべて手続き的に生成する(外部3Dモデルを持たない)。
既存3Dゲーム2本が、依存の有無で異なる方式を既に実証している。

| 既存ゲーム | サイズ | 方式 | 依存 |
| --- | --- | --- | --- |
| `game.html` (ミニRPG) | 22KB | Canvas 2D レイキャスティング。魚眼補正まで実装 | なし |
| `syuuthingu.html` (ゾンビシューター) | 72KB | three.js r128。`SpotLight` + `FogExp2` + `MeshStandardMaterial` | cdnjs |

**手続き的ジオメトリは単一 HTML 契約と構造的に噛み合う。** アセットファイルが存在しないため、
ライセンス問題もサイズ問題も発生しない。`syuuthingu.html` は刀も銃も `BoxGeometry` と
`CylinderGeometry` の組み合わせで作っている。

three.js は `public/team-t-app/games/vendor/` へ**自己ホスト**し、CDN から読まない。
`syuuthingu.html` が使う cdnjs には、障害時に iframe の `onLoad` だけが発火して
10秒の `load-error` 判定も通らず、黒画面のままコインだけ消えるという実害がある。
自己ホストなら同一オリジン・オフライン動作で、複数ゲームから共有でき、
将来 `syuuthingu.html` の CDN 依存も解消できる。バージョンは既存に合わせて r128 で固定する。

### 訂正記録

この節は当初の記述を2点訂正している。判断の履歴として残す。

1. 当初「外部ライブラリ・CDN に依存しない」と書いたが、これはプロジェクトの実態ではなく
   誤った制約だった。`syuuthingu.html` は既に cdnjs から three.js を読んでいる。
   依存を禁じるのではなく、**自己ホストで依存を可視化・固定する**方針へ改めた。
2. 当初「見た目の底上げに `kenney.nl` の CC0 アセットを使う」と書いたが、これは
   2D スプライト向けの助言であり 3D では不適切だった。Kenney の OBJ/GLTF は
   ローダーライブラリとモデルファイルの別置きを要求し、単一 HTML 契約を壊す。
   3D を選ぶ場合は手続き的ジオメトリを採る。

## 前提(これが崩れたら結論も変わる)

- 見た目の完成度が、単一 HTML・数百KB の範囲で既存8本を明確に超えられる
- `team-t-game-dialog.tsx` の契約(特に同一オリジン検証と起動前課金)が今後も維持される
- 9本目を増やすこと自体に価値がある(既存8本で足りている、という判断は採らなかった)

## 却下した選択肢

| 選択肢 | 却下理由 | どういう状況ならこちらが勝つか |
| --- | --- | --- |
| OSS Unity プロジェクトを WebGL ビルドして同梱 | postMessage 送出に `.jslib` プラグイン + C# 改造が必要。そのために Unity Editor 10〜15GB の導入とバージョン合わせが要る。成果物は20〜60MBでリポジトリを恒久的に肥大させる。目的(見た目の完成度)に一切寄与しない | 目的が「Unity→WebGL ビルドを自分で通す学習」だった場合。今回は目的が異なると確認済み |
| itch.io / Unity Play の iframe 外部埋め込み | `event.origin !== window.location.origin` で構造的に不可能。加えて itch.io は埋め込み側に `Cross-Origin-Opener-Policy` / `Cross-Origin-Embedder-Policy` ヘッダを要求するが、GitHub Pages はカスタムヘッダを設定できない。他者のゲーム埋め込みには作者の許諾も必要 | 同一オリジン検証を緩め、かつ独自ホスティングへ移行した場合 |
| 別リポジトリ + 別 GitHub Pages で配信し iframe | 同上。オリジンが変わる時点で契約違反 | 同上 |
| HexGL (MIT, コード+リソース含む) を fork して改造 | 現行ブラウザでの動作は実機確認済みで見た目の水準は高い(FWA 受賞・Chrome Experiments 掲載)。しかし約15MB の恒久的リポジトリ肥大に加え、内蔵タイトル画面/設定メニューの除去、3周(実測 2'15"〜2'55")から短尺への改修、タイマー、postMessage の実装が必要 | +15MB を許容してでも、自作では届かない映像水準を最優先する場合 |
| js13kGames の上位入賞作を流用 | 下記「既製ゲーム探索の結果」を参照。品質・サイズ・自己完結性は理想的だが、Graphics 上位作はライセンス無記載で再配布不可 | 作者から個別に許諾を得られた場合。特に CLAWSTRIKE は契約適合性が最も高い |
| 何もしない(既存8本のまま) | 9本目を追加したいという要望が起点 | 既存8本の利用状況が伸びておらず、本数より各本の作り込みが課題だと判明した場合 |

## 既製ゲーム探索の結果(2026-07-21 実施)

「作る前に、既に良いものがないか探す」を実施した記録。結論は変わらなかったが、
探索しなければ「もっと良いものがあったのでは」という疑いが残り続けるため記録する。

探索プールは **js13kGames** を採用した。13KB 制限が「完全自己完結・外部依存ゼロ」を
規約で強制しており契約要件を構造的に満たすこと、Graphics が専門審査員による
単独採点項目であり品質基準に直接対応することが理由。14年・約200本/年の蓄積がある。

品質は十分だった。2025年の上位作は既存8本を明確に超える。障害はライセンスだった。

| 調査項目 | 結果 |
| --- | --- |
| js13k 規約の権利関係 | 参加者は**主催者に対してのみ**ライセンスを付与し、参加者が全知的財産権を保持する。第三者には何の権利も渡らない |
| MIT 比率 | GitHub 検索インデックス上、`org:js13kGames` の220本中 MIT は31本(約14%)。いずれも星0〜3の無名作 |
| Graphics 上位作のライセンス | CLAWSTRIKE(4.56)・Cat Survivors(4.55)・Black Cat on Cybertower(4.53)・Infinite Hunger(4.40)・Gravity Cat(4.27) は**すべてライセンス無記載** |
| 唯一の例外 | Witchcat(Graphics 4.51) は MIT。ただし10〜30分の探索アドベンチャーで短時間クリア判定に不適合。加えてピクセルアートが別作者(Lylouf)でアセット権利が別問題になる |
| CLAWSTRIKE 作者(remvst) | 公開10リポジトリすべてがライセンス無記載。失念ではなく方針と判断される |

判明した一般則: **品質 × 再配布可能性 × 契約適合の三条件は、最良のプールを全数調査しても
ほぼ空集合になる**。GitHub で公開されていることは利用許諾を意味しない。

副産物として、品質確保の第三の道が見つかった。「既製ゲームを流用する」と
「ゼロから作る」の二択ではなく、**CC0 のアセットを使って自作する**道がある。
`kenney.nl` は数千点のゲームアセットを CC0(パブリックドメイン相当)で配布しており、
商用利用可・帰属表示不要・統一されたアートスタイルを持つ。
「自作＝見た目で妥協」という暗黙の前提は成り立たない。

## 実装の現在地(2026-07-21 時点)

`neon-tunnel.html` は動作する状態まで到達済み。次に触る人はここから読む。

**完了**

- `public/team-t-app/games/vendor/three.min.js` — three.js r128 を自己ホスト(603KB, MIT)
- `public/team-t-app/games/neon-tunnel.html` — 本体。手続き的ジオメトリのみ、外部モデルなし
- `lib/team-t-app/games.ts` に `neon-tunnel` を登録(cost 3 / maxReward 3)
- 契約検証済み: `{type:'game:ended', coin:3}` の実発火を確認。コイン配分は 3/2/1/0 で maxReward 内
- 難易度は走行シミュレーションで調整済み: 被弾0/1/2/3/4回 → コイン 3/2/2/1/0、4被弾で時間切れ

**未完了**

1. **プレビュー画像 `public/team-t-app/game-previews/neon-tunnel.png` が未作成**。
   他8本は用意済みなので、このままではカルーセルで画像が404になる。
2. **実プレイ未検証**。難易度はシミュレーションの予測であり、遊んだ手応えではない。
   コース上の障害物は約41個で3被弾まで許容 = 回避成功率93%が要求される。厳しすぎる可能性あり。
   調整つまみは3つとも定数として先頭に集約してある: `HIT_DROP`(0.30→0.40 で1被弾のコストが
   3.4→2.5秒)、`TIME_LIMIT`(50→53 で許容被弾4回)、障害物間隔(`i * 78` を広げる)。
3. 見た目はまだ最小限。グロー表現・背景・自機の造形に伸びしろがある。

**設計上の注意(同じ失敗を繰り返さないため)**

独立した失敗条件を複数置くと、必ず一番厳しいものだけが効いて残りは死ぬ。
初版は「時間切れ」と「シールド枯渇」を併設した結果、シールドが先に発動して
制限時間が一度も制約にならなかった(被弾4回でも残り17.9秒あった)。
現在は失敗条件を時間切れのみに一本化し、被弾は速度低下=時間ロスとして効かせている。
ここへ被弾数を戻すと、同じペナルティの二重計上になる。
