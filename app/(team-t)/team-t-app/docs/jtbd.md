# Team T Jobs to Be Done

- Status: approved
- Based on: approved `product-brief.md`
- Primary user: Web APIを学び始めた学生・初学者

## Primary job

Web APIを使った制作や学習を始めたいとき、難しい公式資料を最初から読み込まなくても、自分の興味や課題に合うAPIを見つけ、何ができるかを理解し、実際の操作結果を確かめたい。そうすることで、そのAPIを次の学習や制作に使うか判断したい。

### Primary job story

> Web APIで何か作りたいが候補や使い方が分からないとき、興味や目的からAPIを探し、その場で用途と実行結果を確認したい。そうすることで、試すAPIを自信を持って選び、次の実装へ進みたい。

## Job map

| Stage | User job | Desired outcome |
| --- | --- | --- |
| Orient | このサイトで何ができ、どの程度試せるか把握する | 177ページ・200 APIの規模と、検索・カテゴリ・おすすめの入口がすぐ分かる |
| Define | 興味、制作目的、知りたい分野を言葉またはカテゴリにする | API名を知らなくても探索を始められる |
| Locate | 条件に合う候補を絞る | 検索結果や分類から少数の候補へ到達できる |
| Compare | 候補の用途、必要条件、利用状態を比較する | live、API key、CORS、fallback、unavailableを見分けられる |
| Try | 入力や操作を行い、レスポンスまたは結果を見る | 成功・失敗・サンプルの違いが理解できる |
| Evaluate | 学習・制作に使う価値があるか判断する | 何ができ、次に何を調べるべきか説明できる |
| Continue | 別のAPIを探す、選択状態を共有・復元する | 探索を中断せず次の候補へ進める |

## Supporting jobs

### Functional jobs

- API名を知らなくても、分野や自然なキーワードから探す
- おすすめから、自分では検索しなかったAPIに出会う
- APIが利用できない理由と、代替となる確認方法を知る
- 選択中のAPIをURLで共有し、後から同じ状態へ戻る
- テーマや表示を端末上で自分に合う状態へ調整する

### Emotional jobs

- APIを試すことへの難しさや失敗への不安を減らす
- 小さな操作でも学習が進んだ感覚を得る
- 多数のAPIから選ぶ際に迷子にならず、自分で選べたと感じる

### Social jobs

- 授業やチーム内で、面白いAPIを短いURLで紹介する
- APIを使った制作アイデアを他者へ説明する材料を得る

## Reward-loop job

### Job statement

APIをいくつか試して探索の区切りに達したとき、自分の進捗と得たコインに気付き、短いゲームを選んで遊びたい。そうすることで達成感を得て、次のAPI探索へ戻りたい。

### Discoverability requirements

- 初回画面で、ゲームの存在、コインの用途、API操作でコインが得られることを一度で理解できる入口を置く
- 所持コインと探索進捗はAPI閲覧を妨げない位置で常時確認できる
- 初回の進捗増加時とコイン獲得時に、ゲームで使えることを短く通知する
- ゲーム入口は常設し、残高不足時にはAPIを試すという回復行動を示す
- ゲーム終了後は結果だけで終わらず、API探索へ戻る明確な導線を置く
- 強い導線は視認性と次の行動の明確さで実現し、強制モーダル、閲覧ロック、過剰な反復通知にはしない

### Guardrails

- ゲームを利用しなくてもAPI探索・試行・共有を完了できる
- ゲーム報酬のための無意味な連打を主要行動として促さない
- V1の進捗は実装難易度を抑えるためiframe内の対象操作クリックで計測するが、APIの成功・理解・習熟度とは表現しない
- コインやゲーム状態は端末内データであり、アカウント資産や金銭的価値を持つように表現しない

## Constraints and barriers

- 初学者はAPI名、HTTP、JSON、CORS、認証の用語を知らない可能性がある
- 外部APIは停止、仕様変更、rate limit、CORS、API keyにより動かない場合がある
- 200 APIは選択肢過多を起こすため、最初から全件を平坦に見せない
- iframe内の既存ページは操作方法と品質が不均一
- GitHub Pagesでは本物のserver authenticationを提供できない
- 個人化を行う場合は、同期されない「端末内プロフィール」と明示する

## Outcome signals

| Outcome | Signal |
| --- | --- |
| 探索開始が容易 | 初見ユーザーが検索・カテゴリ・おすすめのいずれかを迷わず選べる |
| 候補到達が速い | 2分以内に関心のある紹介ページを1件開ける |
| API理解が進む | ユーザーが選んだAPIの用途と必要条件を説明できる |
| 試行結果が明確 | 成功、外部制約、sample fallbackのどれかを判別できる |
| ゲームに気付ける | 初回セッションで、ゲームの存在とコイン獲得方法を説明できる |
| ゲームが任意である | ゲーム未利用者もcore flowを完了できる |
| 探索へ戻れる | ゲーム終了後またはAPI評価後に次のAPI選択へ移れる |

## Validation hypotheses

1. 初学者はAPI名検索よりも、カテゴリと用途説明を組み合わせた方が候補を見つけやすい
2. APIの可用性状態を事前表示すると、外部制約による失敗を自分の操作ミスだと誤認しにくい
3. 初回進捗とコイン獲得時の文脈的な案内があれば、強制表示なしでもゲームの存在と利用方法に気付ける
4. 将来検証: クリック数ではなく「意味のあるAPI操作」を進捗にすると、報酬と学習目的がより一致する。この変更はAPIごとの完了条件設計が必要なためV1では見送る

## Human decisions

2026-07-16に以下を決定済み。

1. Primary jobは「APIを発見・理解・試行し、制作や学習に使うか判断する」とする
2. ゲーム導線は、常設入口＋初回進捗時＋コイン獲得時の3箇所を基本とし、強制表示は行わない
3. 意味のあるAPI操作に基づく進捗判定は実装難易度が高いためV1では見送る。V1は既存のクリックベース進捗を維持し、学習達成度とは表現しない
