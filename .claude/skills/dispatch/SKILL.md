# Dispatch — 分解・直接振り分け・検証

目的はトークン消費の最小化。ここでの「最小化」とは (a) メイン context を太らせない、
(b) 大量読み込みを使い捨て context に押し出す、(c) サブタスクごとに安いモデル/effort を使う、
の3つであって「常に委任する」ことではない。振り分けに中継エージェント(dispatcher)は使わない
— brief と結果が中継 context を素通りして二重払いになるだけで、分解済みタスクの振り分けは
決定的作業だから。

## 0. Triage — まず委任すべきか判定する

次のいずれかなら委任せず inline で実行してスキルを終える:

- 対象ファイルが3個以下、または手順が3ステップ以下
- 会話の文脈(直前のレビュー内容・ユーザーの好み等)が作業の本体
- すでに自分が対象ファイルを読み込んでいる

理由: ワーカーは冷起動で会話を知らない。brief 作成+システムプロンプト+ファイル再読込の
固定費があり、小タスクでは inline より確実に高くつく。トークン最小化の第一手段は「委任しないこと」。

## 1. 分解 — 自己完結 brief を作る

結果同士の突き合わせが最も薄くなる境界でサブタスクを切り、各サブタスクを以下のテンプレで
自己完結 brief にする。会話履歴の要約を渡さず、必要な事実だけを列挙する:

```
Goal: <1-2文。何ができたら完了か>
Inputs: <絶対パスを列挙。ワーカーに探索させない — 探索は最も高い操作>
Output: <期待する返答の形式。JSON schema か固定見出し。「返答はこの形式のみ、報告文は不要」と明記>
Done: <完了条件と、あれば検証コマンド>
Don't: <触ってはいけないファイル・やらなくてよいこと>
```

## 2. 振り分け — 直接 dispatch(中継なし)

- **2〜4個の独立サブタスク** → 1メッセージ内で並列に Agent 呼び出し(同時起動)。
- **5個以上、または段階依存(収集→変換→検証など)** → Workflow ツール。
  振り分け・ループ・バリアはスクリプト側(JavaScript)に置けばモデルトークン0で済む。
  `agent(prompt, {schema})` で返答を構造化する。
- agent type の選択:
  - 読むだけ(調査・検索・監査) → `Explore`(read-only で安い)
  - 編集・コマンド実行あり → `general-purpose`
  - プロジェクトに専用 agent 定義があるならそれを最優先
- 右サイズ化: 機械的・定型な作業は `model: haiku` + 低 effort。判定・検証は高 effort。
  迷ったら指定せず継承。
- 並列ワーカーが同じファイル群を編集するときだけ `isolation: worktree`(高コストなので
  読み取り専用や非競合なら付けない)。

## 3. 回収 — 構造化された結果だけ受け取る

ワーカーの最終メッセージは報告文ではなく結果データにさせる。Workflow なら schema で強制、
Agent ツールなら brief の Output 欄で「このJSON/表のみを返す」と指定する。
生ログや経過説明を返させない — 返答トークンはそのままメイン context の重量になる。

## 4. 検証 — 決定的チェック優先

- まずオーケストレータ自身がコマンドで検証する(テスト・lint・validate 等)。
  モデル判定より安くて確実。
- モデルでしか判定できない意味的な正しさだけ検証エージェントに出す。件数が多ければ並列。
- **失敗したサブタスクは新規スポーンしない。** SendMessage で同じ agent に追撃指示を送る
  — ワーカーの context が生きているので、再読込コストを払わずに修正できる。

## 5. 報告

ユーザーには統合結果と検証結果だけを示す。ワーカーの生返答を貼らない。
省略・未達があれば正直に明記する(全部できたように見せない)。

## アンチパターン(やらない)

- dispatcher / 中継エージェントを挟む — brief と結果の二重払い
- 同一内容のワーカー定義を複数作る — 同じ agent type を N インスタンス並列起動すればよい
- 「念のため」ワーカーにリポジトリ全体や CLAUDE.md 群を読ませる — Inputs で必要パスだけ渡す
- ワーカーの自己申告(「できました」)を検証にする — 必ずオーケストレータ側で検証する
- 小タスクを分解して見せ場を作る — Triage で inline に倒すのが正解
