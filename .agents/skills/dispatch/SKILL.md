# 最小トークンでタスクを実行する

手段のはしご。上から順に試し、無理なときだけ降りる。

## ① 計算に置換する(最優先)

答えがコマンド/スクリプトで**算出**できるなら、ファイルの中身を読まずに数値・一覧だけ得る。

- 件数・存在・対応関係の照合 → `grep -c` / `glob` / `node -e '...'` ワンライナー。
- コマンド出力は受け取る前に絞る(`| head`、集計値のみ、失敗分のみ)。
- **「答えは計算できないか?」を毎回最初に問う。** これが最大の節約。

## ② 必要箇所だけ読む

計算では出ず、中身を見て判断するしかないとき。

- `Read` は offset/limit で該当範囲だけ、`Grep` は該当行+前後だけ。ファイル丸読みしない。
- すでにこの会話で読んだ内容・確定した事実は読み直さない。
- 調べ物の途中経過や生ログを長々と引用しない。結論に必要な行だけ扱う。

## ③ 委任する(自分の context が溢れる時だけ)

読む対象が大きすぎて、①②で載せるとその後の作業に響く時だけ。子1体につき冷起動+
再把握で数万トークンの固定費がかかるので、**子が肩代わりする読み込み量がそれを超えない
なら委任は損** — ①②に戻る。時間短縮だけを目的に委任しない。

**brief は自己完結。** 会話要約でなく必要な事実だけ。探索は高いのでパスは列挙して渡す:

```
Goal:   何ができたら完了か
Inputs: 絶対パスを列挙
Output: 返答形式(schema/固定見出し)。「この形式のみ、報告文不要」と明記
Done:   完了条件+あれば検証コマンド
Don't:  触らないファイル・やらないこと
```

- 直接振り分ける。中継エージェント(dispatcher)は挟まない(二重払い)。
- 独立2〜4個 → 1メッセージで並列 Agent。5個以上/段階依存 → Workflow(振り分け・ループは
  スクリプト側でモデルトークン0、`agent(prompt,{schema})` で構造化回収)。
- 読むだけ → `Explore`。編集/実行 → `general-purpose`。定型は `model: haiku`+低 effort。
- **孫スポーン禁止**(固定費が二重になる)。返答は schema か行数上限で強制、生ログ禁止。
- **子の完了はゴールではない。** 通知後、同じターンで統合→検証→成果物出力まで進める
  (止めると再開の追撃コストが無駄払い)。

## 検証 — コマンド優先

- コマンドで裏取り(テスト・lint・件数照合)。モデルによる検証はトークン増の主犯、最後の手段。
- 失敗した子は再スポーンせず SendMessage で追撃(context が生きている)。
- **「0ヒット=対象外」を疑う。** 「全件」と書くなら母集合を glob で取り件数照合する。grep は
  NUL/binary を**エラーでなく0ヒットで黙って飛ばす**ため、非ヒットが「該当なし」か「未走査」か
  区別できない。件数が合わなければ疑うファイルを個別に開き、埋まらなければ「M/N件確認、残り未走査」と書く。

## 報告

- 統合結果と検証結果だけ。省略・未達は正直に明記。
- 委任した場合は完了通知の**実トークン数**を添える(過小評価の罠がある)。

## アンチパターン

- ①②を試さず委任 / 計算で出る答えを目視 / 「念のため」全体を子に読ませる /
  自己申告を検証にする / 小タスクを分解して見せ場を作る
